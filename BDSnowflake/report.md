# Отчёт по лабораторной работе №1

## Тема работы

Нормализация исходных данных в аналитическую модель «снежинка».

## Цель работы

Целью работы было преобразовать исходные данные из CSV-файлов в аналитическую модель данных типа «снежинка» в PostgreSQL, выделив таблицу фактов и таблицы измерений, а также обеспечить корректную загрузку и проверку результата.

## Ход выполнения работы

Для выполнения работы я использовал PostgreSQL в Docker. В `docker-compose.yml` я настроил контейнер с базой данных `petstore`, подключил папки со скриптами и исходными CSV-файлами, чтобы вся инициализация выполнялась автоматически при запуске контейнера.

Сначала я создал промежуточный слой загрузки. Для этого в файле `00_create_staging.sql` были подготовлены таблицы `raw_data` и `raw_data_load_buffer`. Таблица `raw_data_load_buffer` используется как временный буфер для загрузки содержимого каждого CSV-файла, а таблица `raw_data` хранит уже объединённые исходные данные. Дополнительно я ввёл поле `file_id`, чтобы различать записи из разных файлов и избежать конфликтов между одинаковыми идентификаторами из разных источников.

Загрузку данных я реализовал в скрипте `01_import.sh`. В нём последовательно обрабатываются все 10 файлов: каждый файл сначала копируется в буферную таблицу, после чего его строки переносятся в `raw_data` с присвоением номера файла. Таким образом, в исходный слой попадают все 10000 строк из набора данных.

После этого я спроектировал «снежинку». В файле `03_ddl.sql` я создал таблицы измерений `dim_country`, `dim_city`, `dim_category`, `dim_customer`, `dim_seller`, `dim_product`, `dim_store`, `dim_supplier` и таблицу фактов `fact_sales`. Структура модели отражает нормализацию исходных данных: страны вынесены отдельно, города связаны со странами, категория товара хранится в собственной таблице, а покупатели, продавцы, товары, магазины и поставщики представлены отдельными измерениями. Таблица `fact_sales` содержит факты продаж и внешние ключи на все основные измерения.

Особое внимание я уделил формированию ключей. Для покупателей, продавцов и товаров я использовал составной подход: глобальный идентификатор формируется как `file_id * 100000 + source_id`. Это позволяет сохранить связь с исходными идентификаторами и одновременно гарантировать уникальность между файлами. Для магазинов и поставщиков, у которых в исходных данных не было удобного числового ключа, я сформировал идентификаторы на основе хеша от набора бизнес-атрибутов.

Заполнение модели я реализовал в `04_dml.sql`. Сначала я создал временную таблицу `stg_normalized`, в которой выполнил предварительную очистку и нормализацию данных: убрал лишние пробелы, преобразовал строки к числовым и датовым типам, обработал пустые значения и для части полей подставил значение `(unknown)`. Затем из этой таблицы последовательно были заполнены измерения, а после них таблица фактов. Для измерений покупателей, продавцов и товаров я использовал выборку `DISTINCT ON`, чтобы сохранить по одной записи на каждую сущность в пределах файла.

В результате построенная модель содержит отдельные сущности для стран, городов, категорий, покупателей, продавцов, товаров, магазинов и поставщиков, а факты продаж вынесены в самостоятельную таблицу. По структуре это соответствует требованиям модели «снежинка», поскольку часть измерений дополнительно нормализована и не хранит повторяющиеся текстовые значения напрямую.

## Результат работы

В ходе выполнения лабораторной работы я подготовил решение, которое автоматически загружает исходные данные в PostgreSQL, создаёт нормализованную схему и переносит данные в аналитическую модель. В исходном наборе данных используется 10 CSV-файлов по 1000 строк, то есть всего 10000 записей. В результате данные были распределены по таблице фактов продаж и связанным с ней таблицам измерений для стран, городов, категорий, покупателей, продавцов, товаров, магазинов и поставщиков.

Для проверки результата я добавил файл `05_validation.sql`. В нём предусмотрены запросы для подсчёта количества строк в таблицах и проверки ссылочной целостности между `fact_sales` и таблицами измерений. Это позволяет быстро убедиться, что загрузка выполнена корректно и в таблице фактов нет «битых» ссылок.

В ходе лабораторной работы я реализовал полный процесс преобразования исходных данных в аналитическую модель «снежинка». В результате получилась нормализованная структура данных, удобная для дальнейшего анализа и построения отчётности.
